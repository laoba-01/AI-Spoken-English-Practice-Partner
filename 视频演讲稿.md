# AI 英语口语陪练 — 视频演讲稿（3-5 分钟）

---

## 开场（~30 秒）

大家好！今天给大家介绍一个我独立开发的 AI 英语口语陪练平台。

你有没有遇到过这种情况——想练英语口语，但找不到母语陪练？报了外教课，又贵又不方便约时间？这个项目就是为了解决这个问题：**随时随地，打开浏览器就能跟 AI 外教对话**。

它的核心体验是：你按住屏幕说话，AI 马上用流利的英文语音回复你，就跟在跟真人外教聊天一模一样。

---

## 技术架构总览（~45 秒）

先快速看一下技术栈。

后端用 **Go 语言** + **Gin 框架**，选 Go 是因为它的协程模型天生适合高并发实时服务。实时通信用的是 **WebSocket**，前端和后端之间保持一条长连接，支持全双工消息推送。

AI 大脑用的是**火山引擎的豆包 Pro 4.0**，英文能力在国内大模型里排第一，口语纠错非常准。语音识别和合成用的是**阿里云智能语音服务**，英文识别准确率 98% 以上，发音也很自然。

前端这边，**Vue 3 + TypeScript + Vite**，UI 组件库是 Element Plus，状态管理用 Pinia。

数据库是 **MySQL 8.0**，缓存用 **Redis 7.0**，文件存储用**阿里云 OSS 加 CDN 加速**。整套服务通过 **Docker Compose 一键部署**。

整个技术栈全部采用国内服务商，不存在访问国外 API 的网络延迟问题，也符合数据合规要求。

---

## 核心功能演示（~60 秒）

接下来看三个最核心的功能。

**第一，三大场景切换。** 日常对话、商务英语、雅思托福考试模拟。不同场景背后是不同的 System Prompt——日常场景像朋友聊天，考试场景就是严格的雅思考官，连纠错的严格程度都不一样。

**第二，语音对话。** 这是主交互模式。前端录音生成 MP3，通过 WebSocket 发送到后端。后端调用阿里云 ASR 做流式识别，边说边转，延迟控制在 500 毫秒以内。识别出文字后，把对话历史一起发给豆包大模型，豆包用流式输出逐字生成回复。这里做了一关键优化——**TTS 预生成**：当 LLM 生成到第 3 句的时候，就提前触发语音合成，跟 LLM 的后续生成并行跑。这样等 LLM 全部生成完，TTS 可能已经提前合成了大半，用户等待时间大幅减少。

**第三，语法纠错和发音评分。** 每次你说完一句话，后台异步调用 LLM 做纠错分析，把错误分成语法、用词、发音三类，每个错误告诉你错在哪、怎么改、为什么这样改。同时给一个 0 到 100 分的发音评分，绿色是优秀，黄色是一般，红色需要改进。纠错结果通过 WebSocket 异步推送到前端，不阻塞对话主流程。

---

## 工程亮点（~60 秒）

再聊聊几个工程实现上的亮点。

**容错和降级。** 整个系统设计了三层降级：数据库挂了会自动切到内存存储，Redis 挂了就不走缓存直查数据库，OSS 上传失败就存本地文件。LLM 调用还加了重试机制——超时最多重试 2 次，高考内容安全拦截自动重试 1 次，尽量保证用户请求不丢。

**限流保护。** 基于 IP 的滑动窗口限流，每个 IP 每分钟最多 10 次请求，超了就返回 429，防止某个用户打爆服务器。

**Redis 缓存策略。** 会话列表缓存 5 分钟，消息列表缓存 2 分钟，写入数据时主动失效相关缓存，保证数据一致性。

**请求日志中间件。** 每个 HTTP 请求记录方法、路径、客户端 IP、状态码和耗时，方便排查问题。

**Docker 多阶段构建。** 编译阶段用 golang 镜像，运行阶段只复制编译好的二进制文件，最终镜像体积很小，启动速度快。

**Swagger 文档。** 所有 API 接口自动生成交互式文档，打开浏览器就能调试。

---

## 性能指标（~20 秒）

性能方面：ASR 识别延迟小于 500 毫秒，LLM 首字延迟小于 1 秒，TTS 合成延迟小于 2 秒。单台 2 核 4G 的服务器就能支持 100 路以上的并发实时对话。1 万月活用户的月度 API 成本估算在 400 到 550 块钱左右，非常经济。

总结一下，这是一个**全链路实时 AI 口语陪练系统**，从语音输入到 AI 文字回复再到语音播放，整个链路延迟控制在秒级。技术栈全部国产化，部署简单，成本可控。
